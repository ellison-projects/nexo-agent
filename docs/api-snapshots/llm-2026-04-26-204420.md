# NexoPRM Agent API ÔÇö LLM Reference

This file is the single-URL bootstrap for any AI agent or script that needs to
read or write NexoPRM data on behalf of a user. Fetch it at
`https://app.nexoprm.com/agentapi/llm.md` and keep it in context.

NexoPRM is a Personal Responsibility Manager built around four pillars: family,
relationships, health, and ambitions. The agent API lets a trusted external
agent touch every one of those areas for any user in the system.

**How to use this doc:**

- Optimized for deciding *when* and *why* to call each endpoint ÔÇö not for memorizing every field.
- Need an exact request/response shape? Send the call with `X-Nexo-Dry-Run: true` ÔÇö it validates input and returns the would-be payload without persisting.
- Or fetch the full OpenAPI 3.1 spec: `https://app.nexoprm.com/agentapi/openapi.yaml`

## Base URL

```
https://app.nexoprm.com
```

All agent endpoints live under `/api/agent/`.

## Authentication

Every request MUST include two headers (unless noted otherwise on a specific
endpoint).

| Header | Value |
|---|---|
| `Authorization` | `Bearer <AGENT_API_KEY>` |
| `X-Nexo-User` | Target user's email OR numeric id |

`AGENT_API_KEY` is a single static shared secret. It grants god-mode across all
users. `X-Nexo-User` picks which user's data the current request touches ÔÇö the
scoping rule inside every SQL query is `WHERE user_id = <impersonated_user>`,
so a request only ever sees that one user's rows.

### Optional headers

| Header | Effect |
|---|---|
| `X-Nexo-Dry-Run: true` | On write endpoints: validate input and return the would-be payload, but do not persist. Audit log entry is still written as a `read`. |
| `Idempotency-Key` | Reserved for future use. Currently accepted and ignored. |

### Endpoints that do NOT require `X-Nexo-User`

- `GET /api/agent/me` ÔÇö optional header
- `GET /api/agent/users` ÔÇö header is ignored
- `GET /api/agent/audit-log` ÔÇö header is ignored
- `GET /api/agent/snapshots` ÔÇö header is ignored
- `GET /api/agent/snapshots/{id}` ÔÇö header is ignored

### Error shape

All errors return:

```json
{ "error": "human-readable message", "code": "error_code" }
```

Error codes: `unauthorized`, `forbidden`, `not_found`, `validation_error`,
`conflict`, `internal_error`, `missing_user_header`, `unknown_user`,
`service_unavailable`.

### Destructive operations

Every `DELETE` endpoint requires `?confirm=true` in the query string. Missing
the flag returns `400 validation_error`. This is a deliberate guardrail
against a hallucinated tool call wiping data.

### Identifiers

- IDs in responses are strings (bigints serialized as text to avoid JS precision loss).
- Timestamps are ISO 8601 UTC.
- Field names in request bodies match the DB column names exactly (e.g.
  `person_id`, `important_dates`, `pinned_at`) ÔÇö NOT the camelCased shapes used
  by the web UI.

## Example prompts ÔåÆ API calls

This section maps real user utterances to the exact API calls an agent
should make. These are the shapes you'll hit 80% of the time ÔÇö skim them
before diving into the full reference below.

### "What's on my plate?" / "Catch me up" / "Give me a briefing"

Any "situational awareness" prompt ÔÇö a morning check-in, an assistant trying to proactively help, a user asking what they're forgetting ÔÇö should hit the one-call briefing first.

```
GET /api/agent/briefing
```

No parameters. See the "Briefing" section below for the full response shape.

Use this BEFORE firing a bunch of individual `GET` calls when the user opens with an open-ended prompt. After the briefing, drill into specific resources only if the briefing didn't already contain the answer.

### "Add a note for Sarah ÔÇö she mentioned her dad's in the hospital"

Two calls: find the person, then log a moment.

```
GET  /api/agent/people?q=sarah&limit=10
  ÔåÆ pick the right id from the response

POST /api/agent/moments
{
  "person_id": "<sarah_id>",
  "content": "Sarah mentioned her dad is in the hospital."
}
```

The moment is persisted and the AI reminder pipeline runs in the
background. Do NOT also call `things-to-remember` ÔÇö moments already cover
timestamped observations.

If the user says "remember that Sarah prefers texts" (a durable fact, not
a timestamped event), use `POST /api/agent/things-to-remember` instead.

### "Sam likes pineapple pizza" (ambiguous name ÔÇö disambiguate first)

When the user mentions a person by a first name or nickname, always search
first and handle the match count explicitly. **Never guess a person id.**

```
GET /api/agent/people?q=sam&limit=10
```

Then branch on the result:

**0 matches** ÔåÆ ask the user whether to create a new person. If yes:

```
POST /api/agent/people  { "name": "Sam" }
POST /api/agent/moments { "person_id": "<new_id>", "content": "Sam likes pineapple pizza." }
```

**Exactly 1 match** ÔåÆ proceed without asking. Go straight to
`POST /api/agent/moments` with that id.

**2 or more matches** ÔåÆ ask the user which one. Present the candidates
with enough context to distinguish them. Use the fields already returned
by the search response: `name`, and whichever of `email`, `phone`, or
relationship/context is set. Example prompt back to the user:

> I found multiple Sams ÔÇö which one?
> 1. Sam Chen (sam.chen@example.com)
> 2. Sam Johnson (555-0199)
> 3. Sam Rodriguez (kids' soccer coach)

Wait for the user's pick before writing. Once they answer:

```
POST /api/agent/moments
{
  "person_id": "<picked_id>",
  "content": "Sam likes pineapple pizza."
}
```

This same pattern applies to ANY write that needs a person: moments,
things-to-remember, list membership, etc. Search ÔåÆ count
match ÔåÆ disambiguate if needed ÔåÆ write.

### "What do I know about Sarah?" / "Catch me up on Eli before I see him" / "Tell me everything about my mom"

Any "refresh my memory on this person" prompt ÔÇö prepping for a call, a
visit, a birthday, a doctor's appointment ÔÇö should hit the full person
detail endpoint. **This is THE endpoint for learning everything about
someone in one call.** Do NOT chain separate calls to
`/api/agent/moments?personId=`, `/api/agent/things-to-remember?personId=`,
`/api/agent/ai-reminders?personId=`, etc. ÔÇö this one call returns all of
it.

```
GET  /api/agent/people?q=sarah&limit=10
  ÔåÆ pick the right id

GET  /api/agent/people/<id>
```

The response contains everything tied to the person:

- `person` ÔÇö name, contact info, `important_dates` (birthdays, anniversaries), `reminder`, `relationship` context, `topics`
- `moments` ÔÇö up to 100 most recent, each with attached `images`
- `things_to_remember` ÔÇö durable facts (preferences, allergies, context)
- `lists` ÔÇö lists they're on
- `connection_groups` ÔÇö groups they're a member of
- `saved_articles` ÔÇö bookmarked news tied to them
- `ai_reminders` ÔÇö up to 100 most recent follow-ups (all statuses)
- `ai_summary` ÔÇö latest AI-generated relationship context

Use it as the starting point any time a prompt is about a specific
person and you need to ground on context before answering or writing.

### "Link my sister's husband to her" / "Link Eli's new friend Jake" / "Sam and John are a couple"

Linking two people **always** goes through a connection group ÔÇö there is no pairwise link resource. The flow:

1. Resolve both people to ids (`GET /api/agent/people?q=`).
2. `GET /api/agent/connection-groups` and scan for an existing group that already contains one of them or obviously fits (a family group, an "Eli's friends" group, etc.).
3. **If a group fits**, add the other person with an optional `role` drawn from the user's phrasing:

   ```
   POST /api/agent/connection-groups/<group_id>/members
   { "person_id": "<new_person_id>", "role": "spouse" }
   ```

4. **If no group fits**, create one with a sensible name and add both members:

   ```
   POST /api/agent/connection-groups
   { "name": "Sam & John" }

   POST /api/agent/connection-groups/<new_group_id>/members
   { "person_id": "<sam_id>", "role": "spouse" }

   POST /api/agent/connection-groups/<new_group_id>/members
   { "person_id": "<john_id>", "role": "spouse" }
   ```

Role hints to pull from natural language: *"her husband"* ÔåÆ `"spouse"`, *"Eli's friend"* ÔåÆ leave `role` null (the group name "Eli's friends" already says it), *"my sister's son"* ÔåÆ `"child"` (of the parents' group) or `"nephew"` (if that's cleaner in context). When in doubt, leave `role` blank ÔÇö the user can always tap to set it later.

Never reintroduce a directional/pairwise link. Groups are the one way people connect.

### "Add milk to my groceries"

One call, using the `active` shortcut:

```
POST /api/agent/groceries/lists/active/items
{ "name": "milk" }
```

If there is no active list yet, that call returns `404 not_found`. Recover
with:

```
POST /api/agent/groceries/lists            (creates a new active list)
POST /api/agent/groceries/lists/active/items { "name": "milk" }
```

Multiple items at once ÔåÆ one POST per item. No bulk endpoint today.

### "Add 'fix the leaky faucet' to my home list"

One call ÔÇö there's one implicit home list per user, so no parent id needed:

```
POST /api/agent/home-items
{ "title": "Fix the leaky faucet" }
```

With a dated reminder (surfaces in the calendar feed and in the briefing's `today`/`upcoming` buckets):

```
POST /api/agent/home-items
{
  "title": "Swap out furnace filter",
  "scheduled_on": "2026-05-01",
  "schedule_note": "new one is in the garage"
}
```

To tack on context later (model number, receipt details, etc.):

```
POST /api/agent/home-items/<id>/notes
{ "content": "Replacement cartridge: Moen 1224" }
```

Mark it done:

```
PATCH /api/agent/home-items/<id>
{ "done_at": "2026-04-18T15:00:00Z" }
```

### "Remember that I like Reach floss" / "Save this cafe ÔÇö they had great salads"

Anything the user wants to recall later that isn't about a person or an actively-shopping item. Products, places, stray facts ("gate code at Mom's is 1234"). One call:

```
POST /api/agent/stash
{
  "title": "Reach Mint Floss",
  "note": "The thin waxed kind ÔÇö thicker than Glide, glides between tight teeth.",
  "tags": ["health", "costco"],
  "photo_urls": ["https://example.com/floss.jpg"]
}
```

Look it up later with the search param:

```
GET /api/agent/stash?q=floss
GET /api/agent/stash?tag=health
```

Add a photo or an extra tag to an existing entry via PATCH:

```
PATCH /api/agent/stash/<id>
{
  "add_photo_urls": ["https://example.com/another.jpg"],
  "tags": ["health", "costco", "my favorite"]
}
```

Stash is the counterpart to things-to-remember: durable facts that aren't tied to a person. If the user is telling you something about a person ("Sarah prefers texts"), use `things-to-remember` instead.

### "Add trash pickup every Tuesday" / "Remind me to take out the trash at 7pm every Thursday"

Use the recurring-tasks resource for anything that comes back on a cadence ÔÇö not one-off todos (those go in working notes) and not birthdays or anniversaries (those go in a person's `important_dates`).

All-day item (digest email only):

```
POST /api/agent/recurring-tasks
{
  "title": "Take out trash",
  "cadence_unit": "week",
  "cadence_interval": 1,
  "anchor_at": "2026-04-21T00:00:00-05:00",
  "has_time": false,
  "lead_time_days": 1
}
```

Timed item ÔÇö fires an email at the exact moment (America/Chicago). Set `has_time: true` and encode the time of day in `anchor_at`:

```
POST /api/agent/recurring-tasks
{
  "title": "Take out trash",
  "cadence_unit": "week",
  "cadence_interval": 1,
  "anchor_at": "2026-04-23T19:00:00-05:00",
  "has_time": true
}
```

For items where missing a cycle matters (insurance renewal, dentist), add `"escalate_when_overdue": true` so the item stays surfaced instead of quietly rolling forward.

### "I just changed the HVAC filter" / "Done with this month's dentist appointment"

One call ÔÇö the `/complete` action sets `last_completed_at` to today and advances `next_due_at` by one cadence interval. Always prefer this over PATCHing `next_due_at` by hand.

```
GET  /api/agent/recurring-tasks?category=home
  ÔåÆ find the HVAC filter item, grab its id

POST /api/agent/recurring-tasks/<id>/complete
```

No request body.

### "Add 'call the plumber' to my plan"

One call, using the `latest` shortcut:

```
POST /api/agent/working-notes/latest/items
{ "content": "Call the plumber" }
```

Same 404-then-create-then-retry pattern if the user has no working notes
yet (POST `/api/agent/working-notes` first).

### "I just thought of something I need to do"

Random ad-hoc todos belong in the user's current plan as a top-level item
(no heading, no parent). These are surfaced in the UI as "Drafts" ÔÇö a
quick brain-dump zone the user organizes into headings later. All of
these utterances map to the same call:

- "I need to call the dentist"
- "Remind me to renew the registration"
- "Todo: pick up dry cleaning"
- "Don't let me forget to email Jim"
- "Add this to my todo list: ÔÇĞ"
- "Jot down that I should ÔÇĞ"

```
POST /api/agent/working-notes/latest/items
{ "content": "<the thing to do>" }
```

Do NOT create a new working note per random todo, and do NOT try to slot
it under an existing heading unless the user explicitly mentions the
project. Leaving `parent_id` null is the right default.

### "I did the thing ÔÇö check it off"

Two steps: find the item, then flip its `checked`.

```
GET   /api/agent/working-notes/latest
  ÔåÆ scan `items` for a content match

PATCH /api/agent/working-note-items/<itemId>
{ "checked": true }
```

Grocery items work the same way with `PATCH /api/agent/groceries/items/<itemId>`.

### "When is my mom's birthday?"

No write, just a lookup. Same one-call pattern as "what do I know about X":

```
GET /api/agent/people?q=mom
GET /api/agent/people/<id>
  ÔåÆ read person.important_dates for the Birthday entry
```

See **"What do I know about Sarah?"** above ÔÇö the same call returns
moments, things-to-remember, reminders, and everything else on the
person, so you can answer follow-up questions from one fetch.

### "Log that the kids had pasta for dinner"

```
POST /api/agent/food-log
{
  "person_name": "Kids",
  "description": "Pasta with marinara",
  "logged_at": "2026-04-17T18:30:00Z"
}
```

### "Save this photo of Eli to his moments" / "Add this picture of our dinner"

Images are attached by passing public URLs on the same create/update
calls. The server fetches each URL, compresses, and stores it. No
multipart, no base64.

```
POST /api/agent/moments
{
  "person_id": "<eli_id>",
  "content": "Eli's birdhouse at camp",
  "image_urls": ["https://example.com/birdhouse.jpg"]
}
```

Same pattern on `POST /api/agent/meals` (with `image_urls`), `POST
/api/agent/home-items` (single `image_url`), `POST
/api/agent/home-items/{id}/notes` (`image_urls`), `POST
/api/agent/groceries/lists/{id|active}/items` (`photo_urls`), `POST
/api/agent/stash` (`photo_urls`), and
`POST /api/agent/working-notes/{id|latest}/items` (`file_urls` ÔÇö
images OR documents).

To add/remove later use the matching PATCH: `add_image_urls` /
`remove_image_ids` (and the `photo_` / `file_` variants for grocery
and working-note items). See the **Attachments** section below for
full rules.

### "Show me what's overdue on my plan"

```
GET /api/agent/working-notes/latest
  ÔåÆ filter items where checked = false
```

### "Remind me to check in with Sarah next week"

Reminders come from two sources:

1. **Moment-driven (automatic):** if the user is telling you something new
   worth remembering (context about the relationship, what was said, what
   happened), log a moment and the AI analyzer will produce a follow-up
   reminder in the background.

   ```
   POST /api/agent/moments
   { "person_id": "<sarah_id>", "content": "Sarah's dad is in the hospital ÔÇö want to check in next week." }
   ```

2. **Ad-hoc (manual):** if it's purely a one-off task ("remind me to call
   the vet Friday", "buy flowers on the 14th") and there's nothing
   relationship-context-worthy to log, create the reminder directly.
   Use `person_id` when the reminder naturally ties to a person,
   `project_id` when it belongs to a multi-week thread (Mother's Day
   planning, interview prep), or both when it lives in both places
   (e.g. "order flowers for mom" ÔåÆ project = Mother's Day, person = mom).
   Omit either when not relevant.

   ```
   POST /api/agent/ai-reminders
   {
     "person_id": "<sarah_id>",
     "due_at": "2026-04-26T17:00:00Z",
     "message": "Check in with Sarah about her dad",
     "rationale": "User asked for a follow-up after Sarah's dad's surgery last week",
     "notes": "Ask how recovery is going and whether she needs help with meals."
   }
   ```

   Reminder tied to a project (Mother's Day):

   ```
   POST /api/agent/ai-reminders
   {
     "project_id": "<mothers_day_project_id>",
     "person_id": "<mom_id>",
     "due_at": "2026-04-30T15:00:00Z",
     "message": "Order flowers for Mom",
     "notes": "User wants to order online ÔÇö last year they used 1-800-Flowers."
   }
   ```

   Field roles:
   - `message` (required) ÔÇö the canonical reminder text. Shown
     as the headline in the in-app reminder card, the ICS calendar feed
     SUMMARY/DESCRIPTION, and the email subject. Keep it concise and
     self-contained ÔÇö the user should be able to act on it without
     additional context.
   - `rationale` (optional, nullable) ÔÇö short "why this reminder
     exists" context surfaced beneath the headline. Set it whenever the
     reason isn't obvious from `message` alone (origin moment,
     user's underlying goal, time-sensitive context). Omit or send
     `null` when there's nothing meaningful to add.
   - `notes` (optional, nullable) ÔÇö free-form long-form details the
     user will want when the reminder fires (draft message, links,
     checklist). Can be lengthy.

Either way the user sees it via
`GET /api/agent/ai-reminders?status=new&dueBefore=<tomorrow_iso>` or on the
in-app `/reminders` page. Prefer option 1 when there's real context to
remember; option 2 when it's just a task.

### "Note that the interview with Acme went well ÔÇö good signal on the role" / "Add a reflection on the half-marathon plan"

These are multi-week threads ÔÇö they belong on a project, not a moment.

```
GET  /api/agent/projects?tag=interview-prep
  ÔåÆ pick the right project id (or POST /api/agent/projects to start one).

POST /api/agent/projects/{id}/notes
{ "content": "Acme round 2 went well ÔÇö they'd want me owning the platform roadmap.", "kind": "note" }
```

For "lessons learned" content, set `"kind": "reflection"` so the user can filter to reflections-only a year later and pull up the retrospective in seconds. The user can start a reflection mid-flight ÔÇö it doesn't have to wait until the project "ends". (Projects don't really end ÔÇö they sink into a Quiet band based on activity.)

Notes are editable ÔÇö if the user says "rework that last note to sayÔÇĞ", PATCH `/api/agent/projects/{id}/notes/{noteId}` with the new `content`.

**Per-note `tags`** slice a long timeline within a single project ÔÇö useful when one project keeps accumulating across cycles. The canonical case is a perpetual project (e.g. an annual father-son trip): tag the 2026 trip's planning notes `"2026"`, the 2027 trip's notes `"2027"`, and the user can pull up just one cycle's notes without splitting the project. Tag pool is per project (no cross-project rollup), lowercased server-side.

**Tag liberally, but reuse before you mint.** `GET /api/agent/projects/{id}` returns `note_tags` ÔÇö every distinct tag already used on this project's notes, ordered by usage. Read it first; if an existing tag fits, use it (even an imperfect match beats a near-duplicate). Only create a new tag when the note is a genuinely new concept or grouping that none of the existing tags capture (a new yearly cycle, a new sub-thread, a new phase). Tag explosion (`2026-trip` / `trip-2026` / `2026`) makes the user's filter chips useless.

Pass on create or PATCH:

```
GET  /api/agent/projects/{id}
  ÔåÆ response.note_tags: ["2026", "lodging", "destinations", ...]

POST /api/agent/projects/{id}/notes
{ "content": "Looked at Whistler vs Banff for next summer", "tags": ["2026", "destinations"] }

PATCH /api/agent/projects/{id}/notes/{noteId}
{ "tags": ["2026", "destinations", "lodging"] }   // replaces the full set
```

### "What did I learn from my last job search?" / "Pull up everything I wrote about Mother's Day"

```
GET /api/agent/projects?tag=mothers-day       # or tag=interview-prep
  ÔåÆ pick the project(s).
GET /api/agent/projects/{id}
  ÔåÆ `notes` has everything ÔÇö filter client-side by `kind=reflection`
    for the distilled takeaways, or by `tags` for a single cycle of a
    perpetual project (e.g. notes tagged `"2026"`).
```

### "Mark all done reminders for today as handled"

```
GET   /api/agent/ai-reminders?status=new&dueBefore=<tomorrow_iso>
PATCH /api/agent/ai-reminders/<id>  { "status": "done" }   // per id
```

### Defaults to assume

- If the user says "my" / "me" without specifying a user, use the
  impersonated user (the `X-Nexo-User` header is already set by the
  caller).
- "Now" means "omit `created_at`/`logged_at`" ÔÇö the server defaults to NOW.
- "My current <plan|groceries>" = `/latest` or `/active` shortcuts.
- Never DELETE as a best-guess response to an ambiguous prompt. Use
  dry-run (`X-Nexo-Dry-Run: true`) if you're unsure.

## Resources at a glance

| Resource | Purpose |
|---|---|
| `users` | List all users in the system (admin-level). |
| `people` | Individual contacts (family, friends, etc.). |
| `moments` | Timestamped notes about a person. Creating one triggers AI reminder analysis. |
| `things-to-remember` | Key facts to remember about a person. |
| `ai-reminders` | Follow-up reminders. Two sources: `moment` (AI-generated from a moment) and `manual` (agent-created one-offs via POST). Optionally attached to a `person` and/or a `project` ÔÇö when both are set, the reminder shows up in both places. |
| `lists` | Simple lists for organizing people. |
| `connection-groups` | Named groups of people with shared context (e.g. "Kids' parents"). |
| `working-notes` | On-demand plan dumps with items and headings. |
| `areas-of-focus` | Ongoing life themes (3ÔÇô5 at a time). |
| `meals` | Saved meal ideas. |
| `food-log` | Calorie/food entries for the user and their household. |
| `groceries` | Grocery lists and items. One list per user is "active" at a time. |
| `home-items` | Home maintenance/chore items with free-text notes. One implicit list per user. |
| `stash` | Pocket knowledge base for non-person-anchored facts the user wants to recall later ÔÇö products they like, places worth revisiting, stray info. Title + optional note + optional location + loose tags + optional photos. |
| `recurring-tasks` | Recurring life-admin items (trash day, insurance, dentist) with a cadence. `next_due_at` auto-advances when an item is completed via the `/complete` action. |
| `projects` | Multi-week initiatives ÔÇö job searches, holiday planning, training plans. Each has a short `description` tagline plus a longer free-text `overview_notes` (what the project is about ÔÇö goals, background, context); an editable dated timeline of notes (each with a `kind` of `note` or `reflection`, plus optional per-note `tags` for slicing a long timeline ÔÇö e.g. `"2026"` on a perpetual annual project); a flat next-actions checklist; optional pillar / target date / freeform tags; and optional linked people. |
| `audit-log` | The agent's own audit trail. Read-only. |
| `snapshots` | Before-state snapshots captured for every destructive write the agent makes (DELETE/PATCH/PUT). Lets you ask "what did this row look like before I changed/deleted it?". Read-only. |
| `briefing` | One-call situational-awareness payload for the next 14 days and what was finished in the last 7 days. Read-only. |
| `features` | Living product features summary (markdown). Read-only. |
| `reports` | Rollup / data-gap reports ÔÇö e.g. people grouped by `relationship.connectionType` (inner circle, broader, or uncategorized). Read-only. |

## Conventions for list endpoints

Every list endpoint accepts:

- `limit` (default 50, max 200)
- `offset` (default 0)

Some accept resource-specific filters documented below.

## Endpoint reference

A compact map of the surface. ### Lists

Lightweight named rosters for action. The mental model: a list is a bucket I'd scan, filter, or pick a name from ÔÇö no shared notes, no shared dates, no per-member roles. Good examples: *Active babysitters*, *Christmas cards*, *People to invite to the barbecue*. If the user would ever want to attach a shared note or date to the whole collection, that's a connection group, not a list.

| Method | Path | Purpose |
|---|---|---|
| GET/POST | `/api/agent/lists` | List / create. |
| GET/PATCH/DELETE | `/api/agent/lists/{id}` | Detail (includes members) / update / delete. |
| POST | `/api/agent/lists/{id}/members` | Add a person. |
| DELETE | `/api/agent/lists/{id}/members/{personId}` | Remove a person. |

### Connection groups

Connection groups are the **only** way to link people together (couples, families, teams, a kid's friend circle). When the user says "link Sam to John" or "link Eli's friend", find an existing group or create one and add both members ÔÇö there is no pairwise link resource. Each member can carry an optional free-form `role` ("spouse", "child", "parent", "sibling", "coach", etc.) describing their position in that group.

**Groups vs lists** ÔÇö both collect people but solve different jobs. Use a group whenever the collection has shared memory: shared `things_to_remember`, shared `important_dates`, or per-member `role`. Families, couples, and teams belong here. Use a list only for pure action rosters that need none of that. Quick test: *"If I could attach one shared note to this whole collection, would that make sense?"* Yes ÔåÆ group. No ÔåÆ list.

Do not confuse `role` with `person.relationship.label` on the Person row. They're two different axes:

- `person.relationship.label` ÔÇö "Who is Sam to the user?" ÔåÆ e.g. *sister*, *best friend*, *boss*.
- `connection_group_members.role` ÔÇö "Who is Sam within **this** group?" ÔåÆ e.g. *spouse* (in her couple group), *sibling* (in the wider family group).

Set `role` when the user's phrasing gives you a semantic hint ("link my sister's husband to her" ÔåÆ add him with `role: "spouse"`). Leave it `null` when the phrasing is ambiguous ÔÇö the group's name carries the context.

**Member payloads are enriched.** `GET /api/agent/connection-groups/{id}` inlines enough Person context that you usually do **not** need a follow-up `/api/agent/people/{id}` call to recall who someone is:

- Contact ÔÇö `email`, `phone`, `address`
- Global context ÔÇö `relationship` (jsonb with `connectionType` "primary"/"secondary" and free-form description)
- Dates ÔÇö `important_dates` (birthdays, anniversaries, milestones)
- Conversation hooks ÔÇö `topics` (text[])
- Signals ÔÇö `pinned_at` (non-null = user-pinned), `last_activity_at`

Only fall back to `/api/agent/people/{id}` when you need moments, things-to-remember, AI summary, or reminder history.

| Method | Path | Purpose |
|---|---|---|
| GET/POST | `/api/agent/connection-groups` | List (sorted pinned-first, then by updated_at) / create. |
| GET/PATCH/DELETE | `/api/agent/connection-groups/{id}` | Detail / update / delete. PATCH accepts `pinned_at` (ISO string to pin, `null` to unpin). Members are denormalized ÔÇö each includes `id`, `name`, `email`, `phone`, `address`, `order`, `role`, plus `relationship`, `important_dates`, `topics`, `pinned_at`, `last_activity_at` from the Person row so you can recall who they are without a second call. |
| POST | `/api/agent/connection-groups/{id}/members` | Add a person. Body: `{ person_id, role?, order? }`. |
| PATCH | `/api/agent/connection-groups/{id}/members/{personId}` | Set / clear a member's `role`. Body: `{ role: "spouse" }` or `{ role: null }` to clear. |
| DELETE | `/api/agent/connection-groups/{id}/members/{personId}` | Remove a person. |

Pin a group (`pinned_at` is an ISO timestamp, mirrors the lists pattern):

```
# Pin
PATCH /api/agent/connection-groups/42
{ "pinned_at": "2026-04-20T17:00:00Z" }

# Unpin
PATCH /api/agent/connection-groups/42
{ "pinned_at": null }
```

```
# Add Tom to Sarah's couple group as her spouse
POST /api/agent/connection-groups/42/members
{ "person_id": "87", "role": "spouse" }

# Later correct his role
PATCH /api/agent/connection-groups/42/members/87
{ "role": "husband" }

# Clear a role
PATCH /api/agent/connection-groups/42/members/87
{ "role": null }
```

### Working notes

On-demand plan dumps with items (headings + checkboxes, one level of nesting max). Pass the literal string `latest` instead of a numeric id to target the user's most recent working note ÔÇö the intended way to "add a todo to my current plan" without listing first. Returns 404 if no notes exist yet.

Items with no `parent_id` and `is_heading: false` are treated as **Drafts** in the UI ÔÇö a brain-dump zone for todos that haven't been organized into a heading yet. The default for ad-hoc todos is to leave `parent_id` null so they land in Drafts.

Heading items can carry a `notes` string ÔÇö long-form project context the user writes on the dedicated heading page (e.g. open questions, decisions, scratch thinking for one project). When summarizing a plan or answering "what's going on with project X", read `notes` on the relevant heading for the richest context; child items give you the todo list, `notes` gives you the narrative. The field is stored on any item, but only headings expose it in the UI ÔÇö don't write `notes` onto non-heading items.

| Method | Path | Purpose |
|---|---|---|
| GET/POST | `/api/agent/working-notes` | List / create. |
| GET/PATCH/DELETE | `/api/agent/working-notes/{id\|latest}` | Detail (items include `file_count` and `notes`) / update / delete. |
| POST | `/api/agent/working-notes/{id\|latest}/items` | Append an item or heading. Optional `file_urls` attaches images or documents. Optional `notes` sets the heading's long-form notes on creation. |
| PUT | `/api/agent/working-notes/{id\|latest}/reorder` | Rewrite `sort_order` for every item. Body: `{ item_ids: [...] }` ÔÇö full ordered list of every item id in the note. Use to promote a heading to the top or rearrange sections. |
| GET | `/api/agent/working-note-items/{itemId}` | Detail with attached `files`. |
| PATCH/DELETE | `/api/agent/working-note-items/{itemId}` | Update (`checked: true`, `notes: "..."`, etc.) and/or `add_file_urls` / `remove_file_ids`. Send `notes: null` to clear. |

### Groceries

At most one **active** list per user at any time. `POST /groceries/lists` retires the old active list automatically. `{id}` accepts the literal `active` to target the current list.

| Method | Path | Purpose |
|---|---|---|
| GET/POST | `/api/agent/groceries/lists` | List (with `item_count` + `unchecked_count`) / create. |
| GET/PATCH/DELETE | `/api/agent/groceries/lists/{id\|active}` | Detail (includes items ÔÇö each carries `stash_id` if linked to a Stash entry) / update / delete. |
| POST | `/api/agent/groceries/lists/{id\|active}/items` | Add item. Requires `name` OR `stash_id` (passing only `stash_id` copies the stash entry's title into `name` automatically ÔÇö see "Linking to Stash" below). Optional `photo_urls` attaches photos. |
| PATCH/DELETE | `/api/agent/groceries/items/{itemId}` | Update (`checked: true` etc.) and/or `add_photo_urls` / `remove_photo_ids`. `stash_id` is set at create time and immutable here. |

**Linking to Stash.** **Before creating any grocery item, search Stash first ÔÇö every time, not just when you think there might be a match.** The user maintains a pocket knowledge base of favorite products tagged `grocery` (e.g. *"up&up Floss Picks (Target)"*). Skipping the lookup and creating a plain row loses the user's curated brand preference.

```
GET /api/agent/stash?tag=grocery&q=<item name>
```

If a match comes back, **pass only `stash_id` and leave `name` null** ÔÇö the server fills the new item's name from the stash entry's title. This is the default pattern: don't re-echo the stash title in `name` unless the user explicitly asks for a different label.

```
POST /api/agent/groceries/lists/active/items
{ "stash_id": "42" }
```

Only pass an explicit `name` alongside `stash_id` when the user wants a label that differs from the stash title ÔÇö e.g. a shorter aisle-friendly *"floss"* instead of the full *"up&up Floss Picks (Target)"*. Either `name` or `stash_id` must be present; passing neither is a 400. The UI follows the same convention: when the user picks a stash suggestion, the client sends `{ stash_id }` only and the server fills the name.

Linking renders the stash's first photo inline on the row and taps through to the full stash detail in the aisle. If no stash match exists, just omit `stash_id` and send `name` normally. The link is immutable after create; if the stash entry is later deleted, `stash_id` becomes null and the grocery row keeps its snapshot name.

### Stash

Pocket knowledge base for non-person-anchored facts. Every entry has a `title`, optional `note`, optional `location`, a `tags` array (free-text), and zero or more photos. Photos are attached via `photo_urls` on create and `add_photo_urls` / `remove_photo_ids` on update ÔÇö same pattern as grocery item photos.

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/agent/stash?q=&tag=&limit=&offset=` | List. `q` matches title/note/location (case-insensitive LIKE). `tag` is exact-match on one tag. Ordered by `updated_at DESC`. Each row includes `photo_count`. |
| POST | `/api/agent/stash` | Create. Required: `title`. Optional: `note`, `location`, `tags` (string array), `photo_urls`. |
| GET | `/api/agent/stash/{id}` | Detail with attached `photos`. |
| PATCH | `/api/agent/stash/{id}` | Update any subset of `title`, `note`, `location`, `tags`, and/or `add_photo_urls` / `remove_photo_ids`. Tags replace the full array ÔÇö send the full desired list. |
| DELETE | `/api/agent/stash/{id}` | Cascades to photos and R2 objects. `?confirm=true` required. |

### Home items

One implicit home list per user (no parent id needed). Open items (`done_at IS NULL`) are returned by default; pass `?done=true` to include completed. Items carry free-text notes as a sub-resource.

Open items can optionally carry a dated reminder: `scheduled_on` (`YYYY-MM-DD`) and a short `schedule_note`. Both are accepted on `POST /api/agent/home-items` and `PATCH /api/agent/home-items/{id}` ÔÇö pass `null` (or `""`) to clear. Scheduled items surface in the user's calendar feed as an all-day event with a morning-of alarm; completing an item (`done_at` set) drops it from the feed.

| Method | Path | Purpose |
|---|---|---|
| GET/POST | `/api/agent/home-items` | List (with `note_count`, `scheduled_on`, `schedule_note`) / create. Optional `image_url` (single URL) sets the primary image. |
| GET/PATCH/DELETE | `/api/agent/home-items/{id}` | Detail (notes each include their `images`) / update (`title`, `done_at`, `scheduled_on`, `schedule_note`; `image_url`: string replaces, `null` clears) / delete. |
| POST | `/api/agent/home-items/{id}/notes` | Append a note. Optional `image_urls` attaches images. |
| PATCH/DELETE | `/api/agent/home-item-notes/{noteId}` | Update (`content` and/or `add_image_urls` / `remove_image_ids`) / delete. |

### Recurring tasks

Recurring life-admin items (trash day, insurance renewal, dentist, HVAC filter, "take out trash at 7pm every Thursday") with a flexible cadence (every N `day`/`week`/`month`/`year`), an optional time of day, and an auto-advancing `next_due_at`. Full CRUD is available, plus a dedicated `/complete` action that marks an item done and advances its next occurrence in one call.

Each row includes `title`, `notes`, `cadence_unit`, `cadence_interval`, `anchor_at` (ISO 8601), `next_due_at` (ISO 8601), `has_time` (true = timed occurrence, fires at-time email; false = all-day, digest only), `lead_time_days` (how many days before `next_due_at` reminders start in the daily digest), `escalate_when_overdue` (true = stays surfaced red after the due date; false = a nightly job quietly rolls `next_due_at` forward past now), `category` (free-text), `last_completed_at`, and `last_reminded_at` (timestamp the at-time cron last pinged this occurrence; cleared on complete).

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/agent/recurring-tasks?due_before=&due_after=&category=` | List, sorted by `next_due_at` ascending. All three filters are optional; `due_before` / `due_after` expect ISO 8601 timestamps; `category` is a case-insensitive exact match. |
| POST | `/api/agent/recurring-tasks` | Create. Required: `title`, `cadence_unit`, `cadence_interval`, `anchor_at`. Optional: `notes`, `has_time` (default false), `lead_time_days` (default 3), `escalate_when_overdue` (default false), `category`. `next_due_at` is seeded to `anchor_at`. |
| GET/PATCH/DELETE | `/api/agent/recurring-tasks/{id}` | Detail / partial update / delete. PATCHing `next_due_at` also clears `last_reminded_at` so the at-time cron fires for the new occurrence. |
| POST | `/api/agent/recurring-tasks/{id}/complete` | Mark done: sets `last_completed_at` to now, clears `last_reminded_at`, and advances `next_due_at` by one cadence interval from the later of the current `next_due_at` and now (preserving wall-clock time for timed items). Use this instead of PATCHing `next_due_at` manually ÔÇö it's the only path that guarantees the same advancement logic the web UI and nightly job use. |

Prefer the briefing's `recurring_tasks` bucket for "what's overdue / due today / due soon / recently completed" ÔÇö it already slices the data into the four buckets an agent usually wants. Drill into this list endpoint only when you need to filter by category or look beyond the 14-day briefing window.

### Projects

Multi-week initiatives (think: a job search, Mother's Day planning, a training plan). Each has a short `description` tagline plus a longer free-text `overview_notes` (goals, background, context ÔÇö what the project is about; distinct from the dated `project_notes` timeline below); an editable collection of dated **notes**; a flat **next-actions** checklist; optional pillar tag; optional target date; freeform tags; and linked people.

**Mental model:** projects are for the stuff that has a lifespan ÔÇö opens, accumulates notes + actions over weeks or months, quietly fades. If the user describes a thread that will keep going ("I want to track my job interviews", "planning Mother's Day", "kid's semester"), start a project. One-off events go in a moment.

Each note carries a `kind` ÔÇö `note` (default, a regular observation) or `reflection` (distilled learning). Notes are fully editable after creation ÔÇö the user reworks a thought over time. When a user asks "what did I learn fromÔÇĞ", filter notes by `kind=reflection`.

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/agent/projects?tag=&include=archived` | List. Excludes archived by default. `?tag=mothers-day` filters case-insensitive. |
| POST | `/api/agent/projects` | Create. Required: `title`. Optional: `description` (short one-line tagline), `overview_notes` (single long-form overview of what the project is about ÔÇö goals, background, context; distinct from the dated `project_notes` timeline), `pillar` (`family` / `relationships` / `health` / `ambitions`), `target_date`, `tags`. |
| GET | `/api/agent/projects/tags` | Distinct tags across non-archived projects with counts. Use for autocompletion / discovery. |
| GET | `/api/agent/projects/{id}` | Full detail in one call ÔÇö `project`, `note_tags` (every distinct tag used across this project's notes; read this before tagging a new note to encourage reuse), `notes` (each with its `images` nested inline), `actions`, `reminders` (open first, then handled), `linked_people`. |
| PATCH | `/api/agent/projects/{id}` | Partial update. Writable: `title`, `description`, `overview_notes` (single long-form overview; pass `null`/`""` to clear), `pillar`, `target_date`, `tags`, `pinned_at`, `archived_at`. Pin/unpin by setting `pinned_at` to an ISO string or `null`. Archive/unarchive by setting `archived_at` to an ISO string or `null`. No dedicated toggle route. |
| DELETE | `/api/agent/projects/{id}` | Cascades to notes, actions, images, reminders. |
| POST | `/api/agent/projects/{id}/notes` | Add a note. Required: `content`. Optional: `kind` (`note` default, or `reflection` for distilled lessons), `tags` (per-note labels scoped to this project ÔÇö see [Per-note tags](#per-note-tags) below), `image_urls`, `created_at`. |

**When to use `reflection` vs `note`:** note = "what happened" (the raw observation, written in the moment). reflection = "the lesson" (distilled, written either at the end OR mid-flight). When the user is looking back across time, an agent filters to reflections only.

**Reminders on projects** ÔÇö projects can carry their own dated reminders for the multi-week thread (e.g. "order flowers" on the Mother's Day project, "send thank-you notes" on the interview-prep project). Create one with `POST /api/agent/ai-reminders` and set `project_id` on the body; it then surfaces both on the project detail page and in `GET /api/agent/projects/{id}` under `reminders` (open first by `due_at`, then handled). When a reminder is naturally about a person *and* belongs to a project, set both `person_id` and `project_id` ÔÇö the reminder shows up in both places. Use `GET /api/agent/ai-reminders?projectId=<id>` to query a project's reminder history outside the detail call.

**Tag handling:** tags are stored lowercase; whatever case the agent writes (`"Mothers-Day"`, `"mothers-day"`, `"MOTHERS-DAY"`) collapses to `"mothers-day"` on save, and duplicates (case-insensitive) are deduped. The `?tag=` filter lowercases the input too, so case doesn't matter on read either.

#### Per-note tags

Two tag axes live on a project ÔÇö don't conflate them:

- **Project-level `tags`** (on `projects.tags`) ÔÇö describe the project itself. Filterable with `GET /api/agent/projects?tag=`.
- **Per-note `tags`** (on each `project_notes` row, exposed as `tags` on `ProjectNote`) ÔÇö slice the note timeline *within* one project. Useful when a single project keeps accumulating across cycles. Pool is per project (no cross-project rollup); same lowercase-on-write rules.

Canonical use case: a perpetual annual project (a yearly father-son trip, a recurring training block). Tag each cycle's planning + reflection notes with the year (`"2026"`, `"2027"`) and the user can filter the timeline to one cycle without splitting into multiple projects.

**Tag liberally ÔÇö but reuse before you mint.** `GET /api/agent/projects/{id}` returns `note_tags`: every distinct tag already used on this project's notes, ordered by usage count desc. Always read it before adding tags to a new note and prefer an existing tag when one fits. Only invent a new tag when the note represents a genuinely new concept or grouping that none of the existing tags capture (a new yearly cycle, a new phase, a new sub-thread). Near-duplicates (`2026-trip` / `trip-2026` / `2026`) defeat the whole point of the filter chips.

Read: per-note tags arrive inline on `GET /api/agent/projects/{id}` under each entry in `notes[].tags`. The aggregate list lives at the top level as `note_tags`. There's no `/notes?tag=` endpoint ÔÇö filter client-side from the inline data.

Write: pass `tags: string[]` on `POST /notes` (initial set) or `PATCH /notes/{noteId}` (replaces the full set; `[]` clears).

### Briefing

One-call situational-awareness payload covering the next 14 days looking forward and the last 7 days looking backward for "what did I finish recently?". Read-only, no parameters.

#### `GET /api/agent/briefing`

**When to call it:** any time the user's prompt is open-ended and you need a grounded picture of their life before responding. Cheaper and less hallucination-prone than firing a bunch of individual reads.

Typical triggers:

- *"Catch me up"* / *"Give me a morning briefing"* / *"What's on my plate this week?"*
- *"What did I get done this week?"* / *"Summarize what I've finished recently"* ÔÇö the briefing is designed to answer this in one call; don't piece it together from individual resource reads.
- *"Am I forgetting anyone / anything?"*
- *"What's going on with the house / groceries / meals this week?"*
- You're about to suggest something proactive and want to make sure it doesn't conflict with what's already on the user's plate.

**What's inside:**

- **Framing** ÔÇö the four app pillars, the user's goals, and their trigger-list keywords
- **Upcoming important dates** ÔÇö next 14 days (birthdays, anniversaries, etc.)
- **AI reminders** ÔÇö `overdue`, `upcoming`, and `recently_done` (last 7 days)
- **Working-note reminders** ÔÇö `upcoming` (coming due in the next 14 days) and `recently_done` (fired/dismissed in the last 7 days)
- **Working notes** ÔÇö every note's full contents (items checked + unchecked with `checked_at` timestamps, per-item reminder info, and the note's `priorities_text`)
- **Recently completed working-note items** ÔÇö flat list of items checked off in the last 7 days with parent-heading context (`content`, `parent_content`, `checked_at`, `note_id`) so you can say "you finished X under project Y on Tuesday"
- **Things-to-remember** ÔÇö durable facts about people
- **Recent moments** ÔÇö last 7 days
- **Pinned people** ÔÇö each with their most recent moment
- **Stale people** ÔÇö not touched in 30+ days
- **Pinned lists** ÔÇö with member counts
- **Connection groups** ÔÇö all groups with their `things_to_remember` and important dates
- **Food log** ÔÇö last 7 days
- **Groceries** ÔÇö `active` (unchecked on the active list) and `recently_done` (checked off in the last 7 days, with `checked_at`)
- **Home items** ÔÇö open items bucketed by schedule state (`overdue` / `today` / `upcoming` / `unscheduled`), plus `recently_done` (last 7 days)
- **Recurring tasks** ÔÇö items bucketed by `next_due_at` relative to today (`overdue` / `today` / `upcoming` within 14 days), plus `recently_done` (items completed in the last 7 days, keyed off `last_completed_at`). An item can appear in both a schedule bucket and `recently_done` ÔÇö completing one sets `last_completed_at` but also advances `next_due_at` to the next occurrence.
- **Meal plans** ÔÇö next 14 days

**How to summarize "what I got done":** collect everything with a `*_at` completion timestamp in the last 7 days ÔÇö `ai_reminders.recently_done[].updated_at`, `working_note_reminders.recently_done[].updated_at`, `recently_done_working_note_items[].checked_at`, `grocery_items.recently_done[].checked_at`, `home_items.recently_done[].done_at`, and `recurring_tasks.recently_done[].last_completed_at` ÔÇö then group by day for a human-readable rundown.

Drill into the dedicated resource endpoints whenever you need more than the briefing contains ÔÇö full person history, older moments, completed items, etc. The briefing is the opening read, not the only one.

### Features

Rollup / data-gap views ÔÇö read-only queries that answer "show me everyone who matches X" without duplicating the main list endpoints. Use these when the user wants a **list**, not a briefing.

Response:

### Snapshots

Every destructive agent write captures a before-state snapshot of the affected
row. For deletes, the cascade-children that would be wiped along with the
parent are captured too ÔÇö e.g. deleting a person also snapshots their moments,
things-to-remember, saved articles, group memberships, etc. Use these to
answer "what did this row look like before I touched it?" without needing a
full restore mechanism.

#### `GET /api/agent/snapshots?userId=&since=&resourceType=&resourceId=&operation=&limit=&offset=`

List response is summary-only (no `row_data` to keep payloads small):

```json
{
  "snapshots": [
    {
      "id": "9001",
      "audit_log_id": "501",
      "user_id": "1",
      "resource_type": "person",
      "resource_id": "42",
      "operation": "delete",
      "related_count": 4,
      "created_at": "2026-04-17T14:30:02Z"
    }
  ],
  "limit": 50,
  "offset": 0
}
```

## Attachments (images and files)

Images and files are attached by passing public URLs inline on the
existing create or update calls. The server fetches each URL,
validates it, compresses (for images), uploads to R2, and links the
stored URL to the parent resource in one transaction. No multipart,
no base64.

### Field names by resource

| Resource | Create field | Update add | Update remove |
|---|---|---|---|
| Moments | `image_urls` | `add_image_urls` | `remove_image_ids` |
| Meals | `image_urls` | `add_image_urls` | `remove_image_ids` |
| Home items (primary image, single URL) | `image_url` | `image_url: "ÔÇĞ"` replaces | `image_url: null` clears |
| Home item notes | `image_urls` | `add_image_urls` | `remove_image_ids` |
| Grocery items | `photo_urls` | `add_photo_urls` | `remove_photo_ids` |
| Stash entries | `photo_urls` | `add_photo_urls` | `remove_photo_ids` |
| Working-note items | `file_urls` (images OR docs) | `add_file_urls` | `remove_file_ids` |
| Project notes | `image_urls` | `add_image_urls` | `remove_image_ids` |

### Rules

- URLs must be `http(s)` and resolve to a public IP. Loopback, RFC1918, link-local, CGNAT, and metadata hosts are rejected.
- Max **10** URLs per call. Max **20** removal ids per PATCH.
- 10 MB cap per fetched file. 15 s total fetch budget per call.
- Allowed mime types: image paths accept `image/jpeg`, `image/png`, `image/gif`, `image/webp`; file paths additionally accept `application/pdf`, Word, Excel, `text/plain`, `text/csv`.
- **All-or-nothing:** if any URL fails validation or fetch, the whole call fails and any partial writes are rolled back ÔÇö no orphaned rows.
- Images are re-compressed server-side (resize to 1600px longest edge, JPEG q=80, EXIF stripped). GIFs and sub-500KB files pass through unchanged.
- `X-Nexo-Dry-Run: true` validates every URL (HEAD + redirect follow) without fetching full bytes or writing anything.

### Errors you'll see

- `validation_error` ÔÇö URL rejected (SSRF, bad mime, oversize, unreachable), or a `remove_*_ids` entry not found on the parent.
- `service_unavailable` ÔÇö image storage not configured on this deployment.
- `internal_error` ÔÇö upstream returned bytes but upload to storage failed.

## Common agent patterns

### Get a one-call briefing ("catch me up", "what's on my plate")

1. `GET /api/agent/briefing` ÔåÆ read everything in one shot.
2. If the user's question is answered by something in the response, answer
   from that data ÔÇö do not fire more reads.
3. If you still need detail (e.g. full moment history for one person), only
   then drill into the dedicated list endpoints.

### Look up and log a moment

1. `GET /api/agent/people?q=sarah` ÔåÆ pick the right id.
2. `POST /api/agent/moments` with `{ person_id, content }`.
3. The AI reminder pipeline runs in the background. Check
   `GET /api/agent/ai-reminders?personId=<id>&status=new` later to see results.

### Quick fact lookup ("when is Sarah's birthday?")

1. `GET /api/agent/people?q=sarah` ÔåÆ get id.
2. `GET /api/agent/people/{id}` ÔåÆ `person.important_dates` contains the answer.

### Link two people ("link Sam to John", "link Eli's friend")

1. `GET /api/agent/people?q=` for each person ÔåÆ pick ids.
2. `GET /api/agent/connection-groups` ÔåÆ find an existing group that fits (already contains one of them, or an obvious match like "Looney Family" / "Eli's friends").
3. **If a group fits:** `POST /api/agent/connection-groups/{id}/members` with `{ person_id, role? }`. Use `role` when the phrasing tells you what the person is within that group ("husband" ÔåÆ `"spouse"`, "son" ÔåÆ `"child"`); leave it null if the group name already carries the context ("Eli's friends").
4. **If no group fits:** `POST /api/agent/connection-groups` with a sensible name, then add each person via `/members` with optional roles.

Never look for a linked-people endpoint ÔÇö there isn't one.

### Mark a reminder handled

1. `GET /api/agent/ai-reminders?status=new` ÔåÆ find the one that's done.
2. `PATCH /api/agent/ai-reminders/{id}` with `{ "status": "done" }`.

### Add a thing to remember

1. Resolve the person.
2. `POST /api/agent/things-to-remember` with `{ person_id, content }`.

### Add an item to "my current plan"

1. `POST /api/agent/working-notes/latest/items` with `{ "content": "..." }`.
2. If the response is `404 not_found`, the user has no working notes yet ÔÇö
   create one with `POST /api/agent/working-notes`, then retry the item
   POST against the new id.

### Check something off the plan

1. `GET /api/agent/working-notes/latest` ÔåÆ find the item by content match.
2. `PATCH /api/agent/working-note-items/{itemId}` with `{ "checked": true }`.

### Promote a heading (or any item) to the top of the plan

1. `GET /api/agent/working-notes/latest` ÔåÆ grab `items` (already sorted by `sort_order`).
2. Build a new ordering: move the heading you want to promote ÔÇö plus every child item that sits under it ÔÇö to the front of the list, keep the rest in their current order.
3. `PUT /api/agent/working-notes/latest/reorder` with `{ "item_ids": [...] }`.
   The array must contain **every** item id in the note (no omissions, no duplicates). Parent/child nesting is untouched ÔÇö only `sort_order` changes.

## Safety notes for agents

- The bearer token is god-mode. Do not log it. Do not echo it back in
  responses to the human.
- Every write is audited ÔÇö if you are unsure whether to proceed, prefer a
  dry run with `X-Nexo-Dry-Run: true`.
- Do not call `DELETE` endpoints without clear user intent. Always include
  `?confirm=true` explicitly in the URL ÔÇö the server rejects the call
  otherwise.
- Prefer `PATCH` over recreating a record. Most updates are idempotent.
- Request bodies use DB column names (snake_case), not the camelCase shapes
  you may have seen in web UI traffic.
